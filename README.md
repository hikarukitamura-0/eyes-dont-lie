## DCON2026開発用リポジトリ
チーム名: 楽園の死者
学校名: 舞鶴高専
作品タイトル: Eyes Don’t Lie　～視線は嘘をつかない～
代表者・氏名: 北村光瑠

授業が理解できず「面白くない」、「つまらない」と感じたり、眠気やスマホにより授業に集
中できない学生のために、授業の集中度を可視化し改善のサポートをするサービスである。ま
た、客観的に授業を評価される機会の少ない教育者や教育機関が学習者の集中度合を把握する
ことで、授業内容の見直しや改善を行うことが可能なサービスである。

---

### 1. ブランチ戦略：Git Flowの簡略版

複雑なGit Flowを避け、**メインブランチ**と**開発ブランチ**を中心に運用します。

* **`main`ブランチ:**
    * **本番環境**にデプロイされる、**常に安定した**状態を保つブランチです。
    * 直接コミットは**厳禁**です。必ず後述のルールでマージします。

* **`develop` ブランチ:**
    * **統合開発環境/ステージング環境**などで使用される、開発の中心となるブランチです。
    * `main` から分岐し、開発タスクはすべてここから個人ブランチとして分岐します。


* **トピックブランチ（作業用ブランチ）:**
    * 新しい機能追加、バグ修正など、**1つのタスクに特化した**ブランチです。
    * 命名規則は「2. ブランチの命名規則」に従ってください。

---

### 2. ブランチの命名規則

作業内容がひと目でわかるよう、接頭辞を付けて統一します。
リファクタリング:コード改善　読みやすくする作業のこと

| 接頭辞 | 意味 | 例 |
| :--- | :--- | :--- |
| `feature/` | 新機能追加 | `feature/user-login-form` |
| `bugfix/` | バグ修正 | `bugfix/api-error-on-checkout` |
| `hotfix/` | 本番の緊急修正 | `hotfix/main-server-crash` |
| `refactor/` | リファクタリング | `refactor/code` |
| `docs/` | ドキュメント関連 | `docs/update-readme` |

---

### 3. コミット、及びコミットメッセージのルール
プルリクにおけるコミット数

* **1PR（プルリク）あたり3〜10コミット**



コミットメッセージは「**何のために、何をしたか**」を明確に記述します。

* **タイトル行（一行目）：**
    **何をしたか**を**簡潔な命令形**で記述します。
    * 例: `feat: ユーザプロファイルページの作成` (機能追加の場合)

* **本文（二行目以降）：**
    * **なぜその変更が必要か**、**どのような影響があるか**を詳細に記述します。（省略可ですが推奨）

* **接頭辞（Type）の推奨：**
英語でなくともOKです。

    | Type | 意味 | 例 |
    | :--- | :--- | :--- |
    | `feat:` | 新機能 (Feature) | `feat: Add payment integration` |
    | `fix:` | バグ修正 (Bug Fix) | `fix: Resolve broken link in footer` |
    | `docs:` | ドキュメントのみの変更 | `docs: Update contribution guide` |
    | `style:` | コードの整形（ロジック変更なし） | `style: Format code with Prettier` |
    | `refactor:` | リファクタリング | `refactor: Rename variable to camelCase` |
    | `test:` | テストの追加や修正 | `test: Add unit test for calculator` |
    | `chore:` | ビルドや環境設定など、本質的でない変更 | `chore: Update dependency versions` |

### 4. マージとコードレビュー (Pull Request)

変更をチームに共有し、品質を保つための必須プロセスです。

1.  **トピックブランチで作業完了後、`develop` に向けて Pull Request (PR) を作成します。**
2.  PRを作成する前に、**最新の `develop` ブランチを自分のブランチに取り込み（`rebase` または `merge`）、コンフリクトを解消**してからPushします。
3.  PRには、**変更の概要**と**確認してほしい点**を記載します。
4.  北村光瑠からの**承認（Review）**を得てからマージします。

* **`main` へのマージ:** `develop` の内容が十分にテストされ、リリース準備が整った段階で、`develop` から `main` へ PR を作成しマージします。

---

### 5. その他の重要な習慣

* **`git pull` は頻繁に:** 作業開始前、コミット前には必ず `develop` や対象ブランチから最新の変更を取り込みます。
* **コミットは細かく:** 「キリの良いところ」ではなく「論理的な区切り」で細かくコミットします。大きな変更を一度にコミットしないようにしましょう（3.コミット、およびコミットメッセージルール参照）。
* **コンフリクトは自分で解決:** `pull` 時やマージ時にコンフリクトが発生したら、**PRを作成した本人が責任を持って解決**します。
